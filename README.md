# A6X Pioneer

A ROS 2 Humble 6-DOF robotic arm with a parallel gripper. The arm mixes three actuator
families on one kinematic chain — Unitree motors (serial), a CyberGear/Robstride motor
(CAN bus), and Dynamixel servos (serial) — unified under `ros2_control` and driven with
MoveIt 2. It also runs in NVIDIA Isaac Sim through the `TopicBasedSystem` plugin.

## Hardware Overview

| Joint | Hardware | Interface | device_id |
|---|---|---|---|
| `shoulder_yaw_joint` | Unitree motor | Serial (`/dev/ttyUSB0`) | 1 |
| `arm_roll_joint` | CyberGear / Robstride motor | CAN bus (`can0`) | 2 |
| `forearm_roll_joint` | Unitree motor | Serial (`/dev/ttyUSB0`) | 3 |
| `lower_wrist_roll_joint` | Unitree motor | Serial (`/dev/ttyUSB0`) | 4 |
| `upper_wrist_pitch_joint` | Unitree motor | Serial (`/dev/ttyUSB0`) | 5 |
| `palm_roll_joint` | Dynamixel servo | Serial (`/dev/ttyUSB1`) | 6 |
| `left_finger_prismatic_joint` (gripper) | Dynamixel servo | Serial (`/dev/ttyUSB1`) | 7 |

The six arm joints are driven by the `arm_controller` (joint trajectory), and the gripper by
the `finger_controller` (gripper action).

## Prerequisites

- **OS:** Ubuntu 22.04
- **ROS 2:** Humble Hawksbill (desktop-full)
- **MoveIt 2:** Humble branch

Install ROS 2 Humble following the [official guide](https://docs.ros.org/en/humble/Installation/Ubuntu-Install-Debs.html).

## Installation

```bash
# Create workspace
mkdir -p ~/workspaces/a6x_pioneer_ws/src
cd ~/workspaces/a6x_pioneer_ws/src

# Clone the repository (this repo is the workspace `src` folder)
git clone git@github.com:NTNU-ERC-Internship/a6x_pioneer.git .

# Install all ROS dependencies via rosdep
cd ~/workspaces/a6x_pioneer_ws
rosdep install --from-paths src --ignore-src -r -y

# Install system tools not covered by rosdep
sudo apt install -y can-utils
```

> **Note:** The `build/`, `install/`, and `log/` directories are **not** committed to git —
> they contain machine-specific absolute paths and are regenerated by `colcon build`. You do
> not create a `build/` folder or run `cmake`/`make` by hand; colcon does that for every
> package. See [Build](#build).

## CAN Bus Setup

The CyberGear/Robstride arm motor communicates over SocketCAN. Bring the interface up after
every reboot (1 Mbit/s):

```bash
cd ~/workspaces/a6x_pioneer_ws/src/cybergear_hardware
sudo bash canbus_init.sh
```

Verify it is up:

```bash
ip link show can0        # should report state UP
```

## Device Ports

Before launching on real hardware, verify the actuators are reachable at the ports the URDF
expects (defaults defined in `a6x_ros2_control/description/urdf/a6x.urdf.xacro`):

```bash
ls -l /dev/ttyUSB0       # Unitree motors (shoulder/forearm/wrist)
ls -l /dev/ttyUSB1       # Dynamixel servos (palm + gripper)
ip link show can0        # CyberGear arm-roll motor
```

If your serial adapters enumerate in a different order, either replug them in the expected
order or add udev rules to pin stable names, then update the `serial_port` / `serial_port1` /
`canbus` properties in the xacro.

## Build

### 1. Unitree Actuator SDK (one-time, manual)

The `unitree_actuator_sdk` under `unitree_hardware/` is a **standalone CMake project**, not a
colcon package — `colcon build` does not build it. Compile it once with the classic
cmake/make flow before building the workspace. This produces the Python binding
(`unitree_actuator_sdk.cpython-*.so`) in its `lib/` folder:

```bash
cd ~/workspaces/a6x_pioneer_ws/src/unitree_hardware/unitree_actuator_sdk
mkdir -p build
cd build
cmake ..
make
```

> Requires `gcc >= 5.4.0` (x86) / `gcc >= 7.5.0` (ARM). The prebuilt vendor libraries
> (`libUnitreeMotorSDK_Linux64.so`, `libUnitreeMotorSDK_Arm64.so`) already ship in `lib/`.

### 2. Build the workspace

```bash
cd ~/workspaces/a6x_pioneer_ws
source /opt/ros/humble/setup.bash
colcon build --symlink-install
source install/setup.bash
```

`--symlink-install` symlinks Python/config files instead of copying them, so edits take effect
without rebuilding. Build a single package with
`colcon build --symlink-install --packages-select a6x_commander`.

## Quick Start

Every session on real hardware follows this boot sequence:

**Step 1 — CAN bus** (after every reboot)
```bash
cd ~/workspaces/a6x_pioneer_ws/src/cybergear_hardware
sudo bash canbus_init.sh
```

**Step 2 — bring up the robot** (ros2_control + controllers + MoveIt)
```bash
ros2 launch a6x_ros2_control a6x_robot.launch.py
```

> For Isaac Sim, skip Step 1 and add `use_sim:=true`
> (`ros2 launch a6x_ros2_control a6x_robot.launch.py use_sim:=true`).
> See [Simulation](#simulation-isaac-sim).

## ros2_control

`a6x_robot.launch.py` starts the controller manager, all hardware interfaces (Unitree +
CyberGear + Dynamixel), the controllers, and `move_group` — sequenced so controllers spawn
after the `joint_state_broadcaster` and MoveIt starts last. It does not open RViz; launch a
display separately if you need one (e.g. `ros2 launch a6x_moveit_config moveit_rviz.launch.py`).

```bash
ros2 launch a6x_ros2_control a6x_robot.launch.py             # real hardware
ros2 launch a6x_ros2_control a6x_robot.launch.py use_sim:=true  # Isaac Sim
```

Verify controllers are running:

```bash
ros2 control list_controllers
```

You should see:
- `joint_state_broadcaster` — active
- `arm_controller` (joint trajectory, 6 joints) — active
- `finger_controller` (gripper action) — active

## MoveIt

Motion planning via RViz interactive markers. On real hardware, planning is already available
from `a6x_robot.launch.py`. To run MoveIt standalone against the demo (simulated) hardware:

```bash
ros2 launch a6x_moveit_config demo.launch.py       # MoveIt + RViz (demo/sim hardware)
ros2 launch a6x_moveit_config moveit_rviz.launch.py # RViz only, against a running move_group
```

Programmatic control (pick-and-place, deterministic IK targets) lives in `a6x_commander`:

```bash
ros2 launch a6x_commander position.launch.py       # move to a target pose
```

## Simulation (Isaac Sim)

The arm runs in NVIDIA Isaac Sim using the `topic_based_ros2_control/TopicBasedSystem` plugin,
which bridges ros2_control to Isaac over topics (`/isaac_joint_command` for commands,
`/isaac_joint_states` for feedback). Simulation vs. real hardware is selected by a single
`use_sim` argument — the same URDF/ros2_control description is used in both cases, so nothing
needs to be hand-edited to switch.

```bash
# Full stack (ros2_control + controllers + MoveIt) against Isaac Sim
ros2 launch a6x_ros2_control a6x_robot.launch.py use_sim:=true

# Or MoveIt standalone (demo defaults to simulation)
ros2 launch a6x_moveit_config demo.launch.py
```

For real hardware, simply omit the flag (`use_sim` defaults to `false`):

```bash
ros2 launch a6x_ros2_control a6x_robot.launch.py        # real hardware
```

## Package Overview

| Package | Description |
|---|---|
| `a6x_description` | URDF/xacro model, meshes, and RViz description launch |
| `a6x_ros2_control` | Top-level launch, controller config, and URDF with ros2_control hardware tags |
| `a6x_moveit_config` | MoveIt configuration (SRDF, kinematics, planning pipeline, Isaac `TopicBasedSystem`) |
| `a6x_commander` | MoveIt commander nodes for pick-and-place and pose control |
| `unitree_hardware` | Unitree motor driver, actuator SDK, and ros2_control plugin (serial) |
| `cybergear_hardware` | CyberGear/Robstride motor driver, CAN bus setup, and ros2_control plugin |
| `dynamixel_hardware` | Dynamixel servo driver, SDK, and ros2_control plugin (serial) |
